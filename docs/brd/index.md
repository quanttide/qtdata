# 量潮数据 - 业务需求文档

## 场景一：数据采集与预处理需求

### 工作角色

- 客户：科研工作者或企业，有数据需求
- 交付方：提供数据采集和预处理服务

### 问题

触发：科研团队开启新课题需要使用外部数据（如GitHub提交数据），但原始数据量大（如几百G的压缩包），需要快速获得清洗后的数据。

现状：客户自己评估需要开发Python脚本，部署到海外云服务器运行，希望一周左右完成。

困难：客户不擅长数据采集和清洗，需要外包，但市场上缺乏专业的学术数据服务商。

后果：项目延期或数据质量不达标，影响科研进度。

### 假设

如果存在这样一个平台，能够快速评估需求，开发数据处理脚本，按时交付清洗后的数据，客户就能专注核心的科研工作。

## 场景二：客户获取与信任建立

### 工作角色

- 获客方：通过特定渠道获取客户
- 客户：高校科研工作者或企业

### 问题

触发：需要持续获取客户。

现状：客户画像是985及以上高校经管科研工作者，科研经费充足（每年几万到十几万），发表压力大。需要把非核心工作外包。

困难：获客渠道有限，主要靠厦大校友圈和知乎大V圈。客户对服务专业性要求高，需要建立信任。

后果：产能不足，无法扩大市场规模。

### 假设

如果存在这样一个平台，能够在学术圈建立影响力，展示专业能力，客户就能主动找上门。

## 场景三：报价与交付规范

### 工作角色

- 商务：报价和项目管理
- 客户：需要透明的价格和规范的交付

### 问题

触发：项目启动前需要报价。

现状：我们根据开发和部署成本定价，分首付款和尾款。客户希望了解项目重难点和甲乙双方分工。

困难：数据处理项目需求经常变动，需要灵活调整报价。不同项目复杂度不同，难以标准化定价。

后果：需求变更导致项目延期或亏损。

### 假设

如果存在这样一个平台，能够透明报价，分阶段付款，规范交付（包括数据集、代码、文档），双方就能建立信任。

---

## 场景四：多业务线管理

### 工作角色

- 运营：管理不同业务线
- 客户：不同领域的需求方

### 问题

触发：业务从科研服务延伸到其他领域。

现状：业务包括量潮科研服务、量潮课堂（大数据培训）、量潮数据咨询（AI企业数据需求）等。不同业务有不同客户画像和交付流程。

困难：业务线增多后，客户管理、项目管理、报价规则需要统一规范。

后果：业务混乱，客户体验不一致。

### 假设

如果存在这样一个平台，能够统一管理客户、项目、报价，就能规模化扩展业务。
